What MenuKit isMenuKit is a UI component library for Minecraft inventory menus.Panels are the composition root. Elements go inside panels. Elements are the things a consumer actually reaches for — buttons, text labels, toggles, slot groups, and whatever else the library grows to include. Consumers compose panels from mixed elements; the library provides the primitives; more specialized widgets stay in consumer mods.Slots are one element type — the structurally complex one, because slots have to integrate with vanilla's sync protocol. The slot infrastructure is heavier than anything else in the library, but conceptually slots are peers with buttons and labels, not the library's reason for existing.Why it existsBuilding UI in Minecraft is harder than it should be. Vanilla's AbstractContainerMenu exposes a flat list of slots and a fragile sync protocol indexed by position in that list. Anything richer than a chest — a panel with buttons and toggles and custom interactive elements — requires threading through vanilla's rendering, input handling, and sync machinery yourself.Worse, the options for where your UI lives are impoverished. You can build a whole custom screen with its own handler, but that's heavyweight for simple cases. You can inject elements into vanilla screens via mixins, but then you're writing rendering and input code from scratch. Vanilla doesn't provide a middle ground.MenuKit is that middle ground. Declare panels of elements; the library handles rendering, layout, input dispatch, sync for slot elements, and composition with the rest of the ecosystem. A mod author builds a panel in the shape they want and the protocol details stay invisible.Two ways to use the libraryBuild your own screen. Register a MenuType, call MenuKitScreenHandler.builder(...), declare your panels and elements, open it. This is the path for standalone UIs — a custom inventory for a mod's block, a configuration screen with toggles and sliders, a crafting-table-alike with slots and custom buttons.Decorate vanilla screens. Write your own mixin into a vanilla screen class and compose MenuKit elements within it. The library gives you Panel, Button, TextLabel, and the rest as primitives; you decide where they render in the vanilla screen's layout. This is the path for adding functionality to existing screens — a sort button on every chest, an info panel in the player inventory, a toggle on shulker boxes.Both paths use the same primitives. The difference is who owns the screen — MenuKit owns the screens you build with the builder; you own the vanilla screens you inject into. The library-not-platform discipline means MenuKit doesn't inject into vanilla screens itself; consumers do that work with their own mixins, composing MenuKit elements where they want them.The hierarchyScreen wraps the handler's lifecycle. It exists on both sides — server and client — registered through ScreenHandlerType the way vanilla expects. The client-side partner is MenuKitHandledScreen, which owns layout computation, rendering, hover detection, keybind dispatch, screen-scoped event dispatch, and drag modes for its own screens.Panel is the fundamental unit of composition. Every element in MenuKit lives inside a Panel, because that's the scope at which visibility toggles, and visibility is a load-bearing concept for the whole dynamic-menu story. A Panel holds an ordered list of elements, a position constraint, a visual style, and a visibility flag.PanelElement is an interface with render, mouseClicked, bounds, and an isVisible hook. Anything that goes in a panel implements it. The library ships a small set of element types as primitives: Button (with styles), TextLabel, and SlotGroup. Consumers implement PanelElement directly for anything more specialized — icon buttons, progress bars, toggles, sliders — or MenuKit grows the set when multiple consumers need the same thing.SlotGroup is the complex element. It carries Storage, InteractionPolicy, QuickMoveParticipation, and the infrastructure that makes slots integrate with vanilla's sync protocol. It looks like the other elements from the outside — it renders in a panel, it participates in layout — but internally it's a lot more machinery. That's fine. One element type being complex doesn't change the framing that elements are the primary concept.MenuKitSlot is a thin subclass of vanilla Slot that carries its coordinates and delegates behavior to its owning SlotGroup. The critical property: a MenuKitSlot is a vanilla Slot to the outside world. Ecosystem mixins into Slot affect MenuKit slots identically.Storage and interaction: the slot element's composabilityBecause slots are the complex element, it's worth explaining how they're composable even inside that complexity.SlotGroup separates two concerns vanilla conflates: where items live and what you can do with them.Storage is a narrow interface: getStack, setStack, size, markDirty. A subinterface PersistentStorage adds serialization. MenuKit ships a small family: PlayerStorage (items follow the player), BlockEntityStorage (items live in a block), ItemStackStorage (items live in a ContainerComponent on another item — how shulker boxes and bundles work), EphemeralStorage (in-memory, drops on close), VirtualStorage (escape hatch with supplier/mutator).InteractionPolicy is a pure behavioral description — predicates and callbacks: canAccept, canRemove, onInsert, onTake, maxStackSize. Named constructors for common cases: input(filter), output(onTake), free(), locked(), display().Because these axes are independent, a full crafting table is just a declarative arrangement: a 3×3 input group with EphemeralStorage, a 1-slot output group with a policy whose onTake consumes inputs, the player's inventory group as shift-click destination. No bespoke handler. Just composition.This is the element system's power applied to the slot case: the complex element is made of composable parts.The declarative topologyThe hardest architectural question was how panels appear and disappear without breaking vanilla's stability assumptions. The answer: they don't. Panels are declared upfront, all of them. The consumer writes the full topology at screen construction; all slots are allocated at OPEN_SCREEN time.What changes at runtime is visibility, not structure. Click a tab, flip a visibility flag, the slots that were always there stop being drawn. The architecture separates cleanly:
Structure — server-authoritative, frozen at construction
Visibility — server-authoritative, mutable via vanilla's clickMenuButton C2S packet
Rendering — client-local, recomputed each frame from the visible structure
Layout is constraint-based. Panels declare a position: BODY stacks vertically, RIGHT_OF / LEFT_OF / ABOVE / BELOW position relative to a named anchor. The layout engine computes bounds each frame from these constraints. Consumers think in relationships, not coordinates.The builder API is declarative:MenuKitScreenHandler.builder(MENU_TYPE)
    .panel("main", p -> p
        .group("container", BlockEntityStorage.of(pos), InteractionPolicy.free())
        .group("player", PlayerStorage.of(player), InteractionPolicy.free())
        .button(/* ... */)
        .text(/* ... */))
    .panel("upgrades", p -> p
        .rightOf("main")
        .toggleKey(GLFW.GLFW_KEY_U)
        .group("slots", EphemeralStorage.of(4), InteractionPolicy.input(isUpgrade))
        .hidden())
    .build(syncId)The builder walks the declaration, allocates slots in flat-index order, populates coordinate maps, produces a configured handler. Declaration order determines flat index order, which determines shift-click priority and tab order naturally.Vanilla-slot substitutabilityThis is the commitment that makes MenuKit a library rather than a platform: a MenuKit slot is a vanilla slot to the outside world. Not wraps one, not adapts one — it inherits from Slot with additive capabilities, and every behavior method either calls super or delegates in a way that preserves vanilla's contract.The consequence is strong. A mod that mixins Slot.canInsert to add a global filter has its filter run on MenuKit slots. A mod that modifies slot rendering renders MenuKit slots the same way. A mod that adds right-click handling to every slot sees MenuKit slots respond identically. None of those mods need to know MenuKit exists.This is the Liskov Substitution Principle taken seriously as a compatibility strategy. MenuKit's additions layer on top of vanilla behavior, never replace it. When MenuKit delegates canInsert to its group's policy, the mechanism is still canInsert, and another mod's mixin into canInsert composes with MenuKit's delegation. The most restrictive answer wins.This is what resolves the "who's responsible for ambient behaviors" question. A consumer who wants custom drag modes on every slot in the game writes a mixin into AbstractContainerScreen or Slot, and substitutability carries that to MenuKit slots. MenuKit doesn't need to deliver ambient behavior — the ecosystem's existing mechanisms do, because MenuKit slots present as ordinary vanilla slots.Inertness"Hidden" has to mean more than "not rendered," because a third-party mod walking handler.slots to consolidate items would otherwise see phantom items — stacks sitting in a hidden panel's slots, invisible to the player but visible to iteration.MenuKit introduces inertness as a first-class state. A slot in a hidden panel is inert to the outside world entirely: canInsert returns false, canTakeItems returns false, getStack returns ItemStack.EMPTY, quick-move skips it, isActive returns false, MenuKitSlot.isInert() exposes the status.The key move is getStack lying. The real stack sits safely in the backing storage, untouched. MenuKit refuses to admit it exists while the panel is hidden. Visibility transitions sync to the client via sendContentUpdates. This is consistent with substitutability — an inert MenuKit slot is a valid vanilla slot that happens to be empty, a legal state for any slot.Elements in hidden panels follow the same rule: not rendered, don't receive clicks, inert alongside their slots.Shift-click as declarative routingShift-click is vanilla's worst API. MenuKit turns it from an imperative "override this method and write a routing loop" into a declarative routing graph.Every SlotGroup declares its QuickMoveParticipation — EXPORTS, IMPORTS, BOTH, NONE. Its existing filters are reused for routing. That's the whole configuration surface.The routing algorithm runs in three priority layers: source-aware baseline (player-backed groups prefer container-backed destinations and vice versa, giving vanilla feel automatically), declared priority (integer on each group, defaults ship sensible), and directional pairing (.pairsWith() edges for furnace-style affinities). A group with no explicit configuration gets vanilla feel because the baseline does the right thing.The dual mode: owned and observedMenuKit's abstractions describe screens it didn't build too — vanilla chests, modded backpacks, anything implementing ScreenHandler.Every Slot carries an inventory reference. A vanilla chest's 63 slots resolve to two distinct Inventory identities: the chest's for the first 27, the player's for the remaining 36. Walk the slot list, group contiguous runs by inventory reference-identity, synthesize a VirtualSlotGroup per run. That handles chests, shulkers, barrels, ender chests, player inventory, horse inventories, and most modded containers.Edge cases — furnace-style handlers where three logical slots share one inventory, brewing stands, bespoke modded handlers — get dedicated recognizers. HandlerRecognizerRegistry is an ordered registry of matchers, each (ScreenHandler) → Optional<List<VirtualSlotGroup>>, tried in order with the inventory-identity walker as fallback. Consumers register their own matchers via public API.Both SlotGroup and VirtualSlotGroup implement SlotGroupLike — the type consumer code programs against. HandlerRecognizerRegistry.findGroup(handler, slot) returns Optional<SlotGroupLike> regardless of whether the screen is native or observed. The uniform abstraction is structural, not aspirational.VirtualSlotGroup's storage is read-only. MenuKit reads through it freely but setStack is a no-op. Consumers wanting to mutate vanilla screens do so through vanilla's own APIs, not through MenuKit. That's the library-not-platform line: MenuKit describes screens it doesn't own, but doesn't mutate them.The line MenuKit chooses not to crossMenuKit stops at being a library. It does not mixin into vanilla's HandledScreen or AbstractContainerScreen. It does not inject behavior into screens it didn't build. It provides recognizers, abstractions, the builder API, and a MenuKit-aware HandledScreen base class for consumers building their own screens. Consumers who want to decorate vanilla screens write their own mixins, using MenuKit's primitives as composition material.This restraint is what substitutability buys. Because MenuKit slots present as vanilla slots, consumers who want ambient behaviors already have a delivery mechanism — the normal modding ecosystem — and MenuKit slots participate automatically. MenuKit doesn't need to be a platform because the platform already exists; MenuKit's contribution is to stay compatible with it.The mixin layer follows the same restraint. MenuKit uses the most additive, most composable mixin patterns — non-cancelling @Injects, MixinExtras @WrapOperation and @ModifyReturnValue, duck interfaces via @Mixin — and avoids @Redirect and @Overwrite entirely. Two mods using MenuKit compose cleanly. MenuKit plus any other well-behaved mod composes cleanly.The contractFive guarantees, empirically verified at the end of Phase 5:
Composability. MenuKit plays well with other mods because it never takes ownership of code paths it doesn't need.
Vanilla-slot substitutability. A MenuKit slot is a vanilla slot. Ecosystem mixins into Slot affect MenuKit slots identically and automatically.
Sync-safety. Structure is decided once and frozen; visibility syncs via vanilla's C2S mechanism; rendering is client-local. The protocol cannot desync.
Uniform abstraction. SlotGroupLike is the type consumers program against; native SlotGroups and observed VirtualSlotGroups both implement it.
Inertness. Hidden is invisible to the world, not just to rendering. Phantom items don't exist.
And five declarative surfaces a consumer configures — three at the group level (Storage, InteractionPolicy, QuickMoveParticipation) and two at the panel level (Visibility, Elements).What the library is forThe mod author who needs to build a custom inventory screen with slots and buttons and maybe a toggle reaches for MenuKit and declares what they want. The protocol stays invisible. The rendering is handled. The shift-click works. The library composes with whatever other mods are installed.The mod author who wants to add a sort button to every chest screen writes a mixin, instantiates a MenuKit Button in their mixin code, and renders it at their chosen position. MenuKit provides the component; the consumer provides the injection. The library doesn't try to be ambient.That's the shape. Library, not platform. Components, not a framework. Slots are one element among many — the structurally complex one, but not the defining one.
