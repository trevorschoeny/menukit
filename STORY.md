STORY.md (interim)What MenuKit isMenuKit is a component library for Minecraft UI.Panels are the composition root. Elements go inside panels. Elements are the things a consumer reaches for — buttons, text labels, toggles, sliders, slot groups, and whatever else the library grows to include. Consumers compose panels from mixed elements; the library provides the primitives; more specialized widgets stay in consumer mods.MenuKit targets three rendering contexts: inventory menus, HUDs, and standalone screens. Panels and elements work across all three. Context-specific machinery (the sync protocol for inventory menus, the tick loop for HUDs, the screen lifecycle for standalone screens) lives in context-specific subclasses beneath the element abstraction. A consumer building a custom inventory screen uses one builder; a consumer building a HUD uses another; a consumer building a standalone screen uses a third. The elements composed inside are the same.MenuKit also ships a small utility subsystem for keybind infrastructure — multi-key keybinds (Alt+Shift+K rather than just K) and the ability for consumer mods to define keybinds in their own config files. This is adjacent to the component library rather than part of it. It's infrastructure that makes Minecraft better for consumers; it doesn't follow component patterns and doesn't compete with them.Why it existsBuilding UI in Minecraft is harder than it should be. Each rendering context has its own quirks — vanilla's AbstractContainerMenu exposes a flat list of slots and a fragile sync protocol; HUDs require custom render callbacks and manual state management; standalone screens give you AbstractWidget and a coordinate grid. Anything richer than what vanilla ships requires threading through low-level machinery yourself, and the patterns don't generalize between contexts.Worse, the options for where your UI lives are impoverished. You can build a whole custom system for one context, but you have to rebuild it for the next. Each context's story is separate. Mod authors end up writing context-specific rendering and input code repeatedly, because there's no shared vocabulary for "a panel of elements" that works across inventory menus, HUDs, and standalone screens.MenuKit is that shared vocabulary. Declare panels of elements; the library handles rendering, layout, input dispatch, and context-specific machinery (sync for inventory slots, tick updates for HUDs, lifecycle for standalone screens). A mod author builds UI in the shape they want, across whichever context they want, and the protocol details stay invisible.Two paths for consumersBuild your own UI. Register the appropriate builder for your context (inventory menu, HUD, or standalone screen), declare your panels and elements, open/show/register as appropriate. This is the path for UIs the consumer fully owns — custom inventories, custom HUDs, custom standalone screens.Decorate vanilla UI. Write your own mixin into a vanilla screen or HUD rendering pipeline and compose MenuKit elements within it. The library gives you Panel, Button, TextLabel, and the rest as primitives; you decide where they render in the vanilla context's layout. This is the path for adding functionality to existing UI — a sort button on every chest, an info panel in the player inventory, a status indicator on the HUD.Both paths use the same primitives. The difference is who owns the rendering context — MenuKit owns the UIs you build with its builders; you own the vanilla contexts you inject into. The library-not-platform discipline means MenuKit doesn't inject into vanilla contexts itself; consumers do that work with their own mixins, composing MenuKit elements where they want them.The hierarchyPanel is the fundamental unit of composition. Every element in MenuKit lives inside a Panel, because that's the scope at which visibility toggles, layout constraints apply, and rendering happens as a coherent group. A Panel holds an ordered list of elements, a position constraint, a visual style, and a visibility flag.PanelElement is an interface with render, mouseClicked, bounds, and an isVisible hook. Anything that goes in a panel implements it. The library ships a small set of element types as primitives; consumers implement PanelElement directly for anything more specialized, or MenuKit grows the set when multiple consumers need the same thing.Context-specific containers wrap the panel system for each rendering context. For inventory menus, MenuKitScreenHandler holds panels alongside the slot infrastructure vanilla requires. For HUDs, MKHudPanel (or its generalized successor) holds panels in the HUD rendering pipeline. For standalone screens, a MenuKitScreen base class holds panels within vanilla's Screen lifecycle. The panel itself doesn't know which context it's in; the containing subsystem handles context-specific machinery.Slot groups: the complex elementSlotGroup is one element type, the structurally complex one. It works only in the inventory-menu context because slots need the sync protocol. Inside inventory menus, SlotGroup carries Storage, InteractionPolicy, QuickMoveParticipation, and the infrastructure that makes slots integrate with vanilla's AbstractContainerMenu.A MenuKitSlot is a thin subclass of vanilla Slot that carries coordinates and delegates behavior to its owning SlotGroup. Critically: a MenuKitSlot is a vanilla Slot to the outside world. Ecosystem mixins into Slot affect MenuKit slots identically. This is substitutability — the library's compatibility commitment — applied to the slot case.The detail matters because slots are where MenuKit's first four phases of migration focused. The slot infrastructure is more complex than any other element in the library. But conceptually, slot groups are peers with buttons and labels — they just happen to require more machinery beneath the element abstraction.The five guarantees (inherited from the inventory-menu work)MenuKit holds five architectural contracts, empirically verified at the end of the inventory-menu migration (Phases 1-5):
Composability. MenuKit plays well with other mods because it never takes ownership of code paths it doesn't need.
Vanilla-slot substitutability. A MenuKit slot is a vanilla slot. Ecosystem mixins into Slot affect MenuKit slots identically.
Sync-safety. Structure is decided once and frozen; visibility syncs via vanilla's C2S mechanism; rendering is client-local. The protocol cannot desync.
Uniform abstraction. SlotGroupLike is the type consumers program against; native SlotGroups and observed VirtualSlotGroups both implement it.
Inertness. Hidden is invisible to the world, not just to rendering. Phantom items don't exist.
These guarantees are specific to the inventory-menu context (they're all about the sync protocol and vanilla slot compatibility). The component-library expansion introduces analogous guarantees for HUDs and standalone screens, but those are worked out as those contexts get generalized. Phase 6's thesis work captures them; this interim story acknowledges they're coming.The line MenuKit chooses not to crossMenuKit stops at being a library. It does not mixin into vanilla's screen classes, HUD rendering pipeline, or input handling. It provides primitives, abstractions, builders, and base classes for consumers building their own UI. Consumers who want to decorate vanilla UI write their own mixins, using MenuKit's elements as composition material.This restraint is what substitutability buys — for inventory menus now, and for HUDs and standalone screens as those contexts generalize. Because MenuKit's elements are additive and don't take ownership of vanilla code paths, consumers get a delivery mechanism for free and the library stays compatible with the rest of the ecosystem.The mixin layer follows the same restraint. Non-cancelling @Injects, MixinExtras operators, duck interfaces. No @Redirect, no @Overwrite.Current state (Phase 5 complete)As of the end of Phase 5:
Inventory menus — fully migrated. The component framework, element system, and slot infrastructure work through the new architecture.
HUDs — MKHudPanel survives from the pre-migration codebase. Not yet generalized under the component-library framing.
Standalone screens — no native support. Consumers building standalone screens currently do so against vanilla's Screen + AbstractWidget directly.
Keybind infrastructure — multi-key keybind support and config-file-defined bindings survive from the pre-migration codebase. Format-agnostic; consumers handle their own config parsing.
Element palette — Button and TextLabel currently ship. More elements are coming as part of the next migration arc.
The next migration arc (Phases 6 and beyond) is a second migration, this time of MenuKit's identity: from "library that handles inventory menus well" to "component library for Minecraft UI." That arc generalizes the component framework across all three contexts, builds out the element palette, and validates the whole thing with consumer mod refactors.What this story will becomeThis story is interim. A full canonical story gets written after the second migration completes — describing the component library as it actually is after contexts are generalized, elements are built out, consumers are validated, and documentation is written.The shape of the eventual story is roughly what's above, but with specifics. Named contexts with their guarantees. Full element palette documented. Decision history captured. The current version is an anchor for the work ahead, not the final document.
